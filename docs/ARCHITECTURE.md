# 架构、信息边界与衍生牌处理

## 数据流

```text
simulator exact state ─┐
HTML/content script ───┼─> TrackerFrame ─> TrackerEngine ─> dashboard / overlay
future vision ─────────┘
```

所有来源都必须先转换成 `TrackerFrame`，再交给同一个 `TrackerEngine`。因此未来接入视觉识别时，可以替换输入适配器，不需要重写记牌账本和前端。

## 公开信息边界

- 我方牌组由房间初始化信息自动绑定。
- 我方手牌、公开角色、公开状态、公开召唤物、骰子和回合标志可以进入账本。
- 对手隐藏手牌和隐藏牌堆只保留数量；`definitionId=0` 永远不会被命名。
- 对手公开打出、公开角色或公开场上的实体才会获得身份。
- HTML/视觉适配器看不见实体身份时提交未知牌；手牌消失不会被擅自解释为“打出”或“弃置”。

这是一条 fail-closed 边界：宁可少记，也不把不可见信息伪造成确定事实。

## 牌账本如何分类

账本不是为几十张衍生牌分别写一条特殊规则，而是根据公开 mutation 的通用生命周期处理：

- 牌实体被创建到牌堆：记录为可见的牌堆生成；
- 牌实体从手牌移动到角色、支援或场上：记录为已打出；
- 实体被移除：根据公开 reason 区分打出、弃置、调和、转移等事件；
- 实体变换：更新已有实体的公开身份；
- 对手隐藏区域的身份：继续遮罩，不因 payload 中偶然出现数字就泄露。

因此像“灵觉者开局洗入三张浮彩”、赤王陵生成禁忌知识、角色天赋牌和效果生成牌，只要上游通知公开了创建、移动或变换事件，就可以走同一套路径；不会因为新增一张衍生牌就必须在 tracker 中新增一个硬编码分支。牌名和卡图由 `data/catalog.json` 与卡图 API 提供。

这仍不等于所有牌的所有条件分支都已被证明。版本更新后需要重新导出目录，并用模拟器探索和 trace 审计覆盖新增机制。

## 项目隔离

- `gi-tcg-tracker` 是独立仓库，不 import、复制或修改 `gi-tcg-robot` 的代码和 harness。
- RL agent 仍在项目外部，通过既有 pybinding 接口工作；tracker 当前不直接控制 agent。
- `genius-invokation` 只作为显式指定的规则和数据上游。
- `references/LumiTracker` 是研究参考，不是运行时依赖。

## 真实原神与模拟器

模拟器可以提供比真实客户端更丰富的精确状态，适合开发和回归测试；真实原神客户端最终只能依赖视觉识别和鼠标/键盘操作。当前项目完成的是模拟器/房间公开 notification 到记牌器的闭环，尚未实现纯视觉识别，也不会向真实原神发送操作。
